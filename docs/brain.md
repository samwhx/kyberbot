# Brain -- Memory Architecture

The brain is KyberBot's long-term memory system. It stores conversations, tracks entities and relationships, extracts structured facts, detects contradictions, generates insights through reasoning, and continuously improves quality through background maintenance.

---

## Overview

The brain consists of six components:

| Component | Purpose |
|-----------|---------|
| **Semantic Search** | Search across all memories by meaning |
| **Entity Graph** | Track people, companies, projects, and relationships |
| **Fact Store** | Structured, verifiable facts with confidence scoring |
| **Timeline** | Temporal log of events and conversations |
| **Reasoning Engine** | Deduction and pattern recognition across entities |
| **Sleep Agent** | Continuous memory maintenance and quality improvement |

All data lives locally in your KyberBot project directory. This fork ships local-only — nothing leaves your machine.

---

## Semantic Search

The semantic search layer provides meaning-based retrieval over all stored memories. When the agent saves a piece of information, it is embedded as a vector and stored locally, enabling retrieval by meaning rather than exact keywords.

### How It Works

1. Agent stores a memory (conversation summary, fact, note)
2. The text is converted to a vector embedding
3. The embedding is stored with metadata (tags, timestamp, priority, tier)
4. When searching, the query is embedded and compared against stored vectors
5. Results are ranked by cosine similarity combined with keyword matching

### Storage

The semantic search engine runs as a Docker container. Data is persisted to `data/chromadb/` in your project directory. The container starts automatically when you run `kyberbot`. Semantic search is optional -- the system continues without it when ChromaDB is unavailable.

### Metadata

Each memory includes:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier |
| `content` | string | The memory text |
| `embedding` | float[] | Vector representation |
| `tags` | string[] | Categorization tags |
| `timestamp` | ISO string | When the memory was created |
| `priority` | float (0-1) | Importance score |
| `tier` | string | `hot`, `warm`, or `archive` |
| `source` | string | Where the memory came from |
| `entities` | string[] | People/projects/companies mentioned |

## Entity Graph

The entity graph tracks discrete entities (people, companies, projects, places, topics) and the relationships between them. It is stored locally at `data/entity-graph.db`.

### Entity Types

| Type | Examples |
|------|----------|
| `person` | Colleagues, friends, family members |
| `company` | Employers, clients, vendors |
| `project` | Active projects, side projects |
| `place` | Cities, offices, venues |
| `topic` | Technologies, interests, goals |

### Features

- **Alias deduplication** -- When creating an entity, the system checks both normalized names and stored aliases. "Nicholas Rith" automatically maps to "Nick Frith" if the latter is already stored as an alias.
- **Noise filtering** -- Conversational noise words (ok, hello, speaker, thing, stuff, someone), CLI commands, file paths, UUIDs, and pronouns are filtered at ingestion time, preventing garbage entities from entering the graph.
- **Typed relationships** -- 15 relationship types: `founded`, `works_at`, `invested_in`, `met_with`, `created`, `manages`, `partners_with`, `located_in`, `discussed`, `related_to`, `reports_to`, `uses`, `depends_on`, `part_of`, `co-occurred`.
- **Access tracking** -- Each entity tracks `access_count` and `last_accessed`, which feeds into decay and tiering decisions.
- **Narrative profiles** -- Entities with 3+ known facts get a natural-language profile generated by the sleep agent. These appear at the top of recall output.

### Entity Profiles

When enough facts accumulate about an entity, the sleep agent generates a concise two-sentence narrative profile:

```
$ kyberbot recall "Nick"

Nick Frith is an AI developer based in Chiang Mai who co-organizes
the weekly Saturday AI meetups and is involved in the CMU curriculum
partnership.

Mentioned 44 times | First seen: Jan 20, 2026 | Last seen: Mar 30, 2026
Type: person
...
```

Profiles are regenerated when the fact count changes, keeping them current without user intervention.

### Querying

```bash
# List all tracked entities
kyberbot recall

# Query a specific entity
kyberbot recall "John"
```

## Fact Store

The fact store holds structured, verifiable statements extracted from conversations. Each fact is a specific claim with metadata: category, confidence, source type, and entity associations. Stored alongside the timeline in `data/timeline.db`.

### Fact Categories

| Category | Examples |
|----------|----------|
| `biographical` | "Caroline is originally from Sweden" |
| `preference` | "Ian prefers async communication" |
| `event` | "Met with Sarah on March 15" |
| `relationship` | "David is CEO of Ohana" |
| `temporal` | "Launch is scheduled for Q2" |
| `opinion` | "Team prefers Next.js over Remix" |
| `plan` | "Migrating to Kubernetes next quarter" |
| `general` | Other factual statements |

### Source Confidence

Every fact and entity mention is tagged with a source type and confidence score reflecting how trustworthy the information is:

| Source Type | Confidence | When Applied |
|-------------|------------|--------------|
| `user-correction` | 1.00 | User explicitly corrects data |
| `user-direct` | 0.95 | User states something in terminal |
| `chat` | 0.85 | Telegram/WhatsApp message |
| `heartbeat` | 0.80 | Heartbeat task output |
| `ai-extraction` | 0.60 | LLM extracted from conversation |

Source confidence is determined automatically from the channel. Terminal `remember` commands get 0.95, messaging channels get 0.85, and AI-extracted facts during sleep or real-time extraction get 0.60.

### Real-Time Fact Extraction

Facts are extracted immediately when a conversation is stored -- not just during hourly sleep cycles. After entity extraction, the system runs a lightweight Haiku call to identify up to 3 concrete facts per conversation and stores them with contradiction detection.

This means `kyberbot recall "John"` works well right after mentioning John, without waiting for the next sleep cycle.

### Fact Lifecycle

```
Conversation Stored
       |
       v
[Real-time Extraction] --> Facts stored (confidence 0.60)
       |
       v
[Sleep Agent: Observe] --> Additional facts extracted
       |
       v
[Contradiction Detection] --> Conflicts resolved or tracked
       |
       v
[Confidence Decay] --> Old unreinforced facts fade (weekly)
       |
       v
[Reinforcement] --> Re-confirmed facts get boosted
```

### User Corrections

When the user says something like "Actually, John works at Beta Inc, not Acme," the system handles it naturally:

1. The correct fact is stored with `user-direct` confidence (0.95)
2. The contradiction detection system compares it against existing facts
3. The old lower-confidence fact is automatically superseded
4. No special commands needed -- the user just speaks naturally

Facts can also be explicitly retracted when the user says something is wrong without providing a replacement.

### Temporal Facts

Facts with time-sensitive content (e.g., "Launch is scheduled for next Tuesday") are automatically assigned an expiry date. Expired facts are swept during the decay step and marked as no longer current.

---

## Contradiction Tracking

When the system detects conflicting facts about the same entity, it handles them based on the confidence gap:

- **Large gap (> 0.3)** -- Auto-resolve: the higher-confidence fact wins, the lower one is superseded.
- **Close gap (<= 0.3)** -- Both facts are kept, and a contradiction record is created. The conflict is surfaced to the user in recall output:

```
$ kyberbot recall "John"
...
Note: I have conflicting info -- "John works at Acme Corp" vs
"John works at Beta Inc". Want to clarify which is correct?
```

When the user clarifies, the correct fact is stored at high confidence and the contradiction is resolved.

## Reasoning Engine

The reasoning engine runs during the sleep cycle, performing deduction and induction on entities with 3+ facts. This is what turns KyberBot from "stores and retrieves" into "understands."

### Deduction (confidence 0.80+)

Derives logically certain conclusions from 2+ existing facts:

- "David is CEO of Ohana" + "Ohana is a Darkstar portfolio company" = "David leads a Darkstar portfolio company"
- "Martin works at Anthropic" + "Ian is Anthropic ambassador" = "Martin and Ian are connected through Anthropic"

### Induction (confidence 0.60-0.75)

Detects probable patterns from 3+ data points:

- "Nick appears at 6 different AI events" = "Nick is a regular in the AI meetup circuit"
- "Entity hasn't been mentioned in 30 days" = "This project may have stalled"

### Quality Detection

The reasoning engine also flags data quality issues: contradictions, likely transcription misattributions, and stale facts that may no longer be true.

### Insights in Recall

Insights appear automatically in recall output:

```
$ kyberbot recall "Nick"

## Insights
- Nick and Martin appear to work together closely -- they co-occur
  at every AI event and both coordinate with CMU [pattern]
- Nick is connected to Ian through both the Anthropic ecosystem and
  the Chiang Mai meetup circuit [inferred]
```

## brain/ Directory -- Markdown Knowledge

The `brain/` directory in your project root stores longer-form knowledge as markdown files. This is for structured information that does not fit neatly into vector search or the entity graph.

**Examples:**

- `brain/projects/payflow.md` -- Detailed project notes
- `brain/people/team.md` -- Team member profiles
- `brain/decisions/architecture-choices.md` -- Decision log

The agent reads and writes files in `brain/` as needed. These files are also indexed for semantic search.

## Hybrid Search

When the agent searches memory, it uses Reciprocal Rank Fusion (RRF) to merge semantic and keyword results without tuned weights:

```
Score = sum( 1 / (k + rank) )  for each result list, k=60
```

Results appearing in multiple search channels (semantic, keyword, temporal) are naturally boosted. Optional Haiku-powered reranking is available for user-facing queries.

### Fact-First Retrieval

For entity-focused queries, the system uses a 5-layer retrieval architecture:

1. **Direct fact search** -- FTS5 + ChromaDB on the fact store
2. **Entity expansion** -- 3-hop BFS traversal through the entity graph
3. **Scene expansion** -- Co-occurring entities and bridge discovery
4. **Supporting context** -- Source conversation segments for each fact
5. **Context optimization** -- Token budget enforcement and deduplication

### Search Commands

```bash
# Basic search
kyberbot search "pricing discussion"

# Filter by entity
kyberbot search "API" --entity "John"

# Filter by time
kyberbot search "meeting" --after "last week"

# Filter by tier
kyberbot search "roadmap" --tier hot

# Fact-first mode
kyberbot brain query "What do I know about John?"
```

## Sleep Agent

The sleep agent is a background process that continuously maintains memory quality. It runs every hour (configurable) and performs a multi-step cycle.

### The Sleep Cycle

#### 1. Decay

Reduces the priority of memories that have not been accessed recently. Repetitive content (heartbeat tasks) decays 3x faster. Pinned items are exempt.

Also performs **weekly fact confidence decay**: facts from AI extraction or chat that haven't been reinforced in 90+ days have their confidence reduced by 5%, with a floor of 0.15. This ensures stale, unreliable information naturally fades.

#### 2. Tag

Refreshes tags on memories older than 7 days using AI analysis, normalizing and enriching tags across time.

#### 3. Consolidate

Merges redundant timeline entries (e.g., repeated heartbeat tasks with the same title).

#### 4. Link

Discovers connections between related memories using Jaccard similarity on tags and entities. Creates edges stored in `data/sleep.db`.

#### 5. Tier

Manages the hot/warm/archive tier system:

| Tier | Criteria | Search Impact |
|------|----------|---------------|
| `hot` | Priority >= 0.65, accessed recently, or highly connected | Boosted |
| `warm` | Priority >= 0.3 or accessed in last 21 days | Normal |
| `archive` | Low priority, not accessed in 30+ days | Penalized |

#### 6. Summarize

Regenerates summaries for memories that have changed tiers. Hot/warm: 5 sentences. Archive: 3 sentences.

#### 7. Observe (Fact Extraction)

Extracts structured facts from conversations that haven't been processed yet. Each fact is stored with category, confidence, entity associations, and temporal expiry detection. Includes contradiction detection against existing facts.

#### 8. Profile

Regenerates the user profile (`data/user-profile.json`) from the fact store when stale. This profile is injected into system prompts for context.

#### 9. Reasoning

Runs deduction and induction passes on entities with 3+ facts (up to 5 entities per cycle). Generates insights stored in the `entity_insights` table. See [Reasoning Engine](#reasoning-engine) above.

#### 10. Entity Hygiene

Cleans up the entity graph:
- Removes transcription artifacts (Speaker 0, Speaker 1, etc.)
- AI-merges variant names (e.g., "Dr. Smith" + "Smith", "Bob" + "Robert")
- AI-merges same-name-different-type duplicates
- Prunes low-value noise entities (1 mention, no relations, 30+ days old)
- Generates narrative profiles for entities with 3+ facts

### Sleep Agent Commands

```bash
# Check status
kyberbot sleep status

# Trigger a cycle manually
kyberbot sleep run

# View memory relationships (edges)
kyberbot sleep edges

# Health check
kyberbot sleep health

# Entity merge audit trail
kyberbot sleep merges
```

## Storage Pipeline

When a conversation is stored (via any channel -- terminal, Telegram, WhatsApp, heartbeat), it flows through:

```
Conversation Input
       |
       v
[Entity Extraction] -- Claude Haiku extracts entities + relationships
       |
       v
[Noise Filtering] -- Blocks noise words, CLI commands, pronouns, paths
       |
       v
[Timeline Storage] -- Temporal event log
       |
       v
[Segment Indexing] -- Split into overlapping chunks for fine-grained search
       |
       v
[Entity Graph] -- Create/update entities, add mentions with source confidence
       |              Alias dedup prevents duplicates
       v
[Real-time Facts] -- Extract 1-3 facts immediately via Haiku
       |              Contradiction detection runs on each fact
       v
[Semantic Indexing] -- ChromaDB embeddings (best-effort)
```

Each subsystem is independently wrapped -- one failure does not block others. The entire pipeline is fire-and-forget.
